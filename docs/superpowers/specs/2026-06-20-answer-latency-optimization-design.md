# 回答链路延迟优化设计

日期：2026-06-20
状态：已确认，待实施

## 1. 背景

当前问答链路已经具备问题理解、受控语料检索、外部搜索、来源治理、结构化生成、引用校验和延伸阅读推荐等能力，但这些能力主要以串行、全量方式执行。

对当前开发版本进行实际测量时，问题“丝柯克是外星人吗”的完整响应约为 27.4 秒：

| 阶段 | 累计时间 | 阶段耗时 |
| --- | ---: | ---: |
| 问题理解完成 | 1.6 秒 | 1.6 秒 |
| LLM 搜索规划完成 | 3.7 秒 | 2.1 秒 |
| 外部搜索与来源审核完成 | 15.7 秒 | 12.0 秒 |
| 正文生成与校验完成 | 19.7 秒 | 4.0 秒 |
| 延伸阅读完成 | 27.4 秒 | 7.7 秒 |

主要重复工作包括：

1. 结构化问题理解已经提供实体、意图和查询词，生成流程仍要求 LLM 再规划一次搜索。
2. 外部搜索会一次性扩展最多八条查询，并对较多候选页面执行发布者抓取和 LLM 内容分类。
3. 正文已经检索到可用来源后，延伸阅读仍会重新执行搜索和页面补全。
4. 轻微格式或表达问题可能触发第二次完整生成，而这些问题可由本地规则处理。
5. 当前接口等待正文和延伸阅读全部完成后才返回最终结果，使非必要工作延迟了正文展示。

本次设计采用“分层快速通道 + 异步补充资料”的混合模式，在不降低回答准确度和可信度的前提下，将普通问答的完整响应控制在 12–15 秒，并尽量在 8–12 秒内展示可信正文。

## 2. 目标

### 2.1 性能目标

- 普通身份、角色、状态和一般剧情问题：正文在 8–12 秒内可见。
- 普通问题的正文与补充资料：在 12–15 秒内全部完成。
- 复杂关系题、深度剧情题或触发修复的请求：正文尽量在 15 秒内可见。
- 已缓存的重复检索应明显快于首次请求，但验收不得只依赖缓存命中。

### 2.2 质量目标

以下保障不得因性能优化而移除：

- 中文问题使用中文回答。
- 回答必须基于当前请求获得的受控语料或外部证据。
- 每个事实段落只能引用实际存在且允许使用的来源。
- 核心实体不得漂移。
- 多实体关系结论必须由同时涉及相关实体的证据支撑。
- 社区分析和推测不得被写成官方确认。
- 剧透控制继续依据玩家的主线完成进度执行。
- 证据不足时返回保守结论，不使用模型记忆猜测。

## 3. 非目标

- 本次不更换 LLM 服务商或搜索引擎。
- 本次不建立长期答案缓存或离线全站爬虫。
- 本次不降低来源治理规则，也不取消规则与 LLM 的互证能力。
- 本次不要求所有深度剧情问题稳定达到普通问答的延迟。
- 本次不对回答页面进行无关的视觉重构。

## 4. 总体架构

回答流程拆分为两个相互独立的阶段。

### 4.1 第一阶段：可信正文通道

第一阶段只执行产生可信正文所必需的工作：

1. 识别问题语言、实体、意图和问题类别。
2. 检索受控语料。
3. 执行分层外部检索，获得足够支撑核心结论的证据。
4. 完成必要的来源审核。
5. 生成正文。
6. 校验语言、实体、引用和权威措辞。
7. 通过 SSE 的 `answer` 事件立即返回正文、正文引用和置信度。

第一阶段的时间预算为 12 秒，复杂请求可延长到 15 秒。证据不足时不得为了满足时间目标提前输出未经验证的结论。

### 4.2 第二阶段：补充资料通道

第二阶段只处理不会影响核心结论的附加内容：

- 延伸阅读；
- 官方视频；
- 更多背景资料入口；
- 未被正文引用的候选来源；
- 低优先级社区来源的发布者核验。

第二阶段优先复用第一阶段的候选来源。只有现有来源不足以形成有价值的推荐时，才执行额外搜索。完成后通过 SSE 的 `resources` 事件追加资料卡片。

第二阶段的默认时间预算为 3 秒。失败或超时不会修改正文，也不会使整个回答显示为失败。

## 5. 问题理解与搜索计划

### 5.1 规则优先

问题理解首先运行本地规则，识别：

- 回答语言；
- 直接主体实体及别名；
- 身份、关系、剧情、当前状态、官方媒体或一般问题等意图；
- 问题分类；
- 初始搜索查询。

规则结果满足以下条件时，直接作为搜索计划：

- 至少识别到一个直接主体实体；
- 查询均保留核心实体或精确别名；
- 意图与问题分类没有明显冲突；
- 不存在多个候选实体无法消歧的情况。

### 5.2 按需调用 LLM

仅在以下情况调用 LLM 辅助理解：

- 规则没有识别到实体；
- 实体或意图存在冲突；
- 别名无法可靠映射；
- 问题结构复杂，规则无法判断主体与背景实体；
- 多实体关系问题无法确定关系双方。

LLM 结果继续与规则结果进行保守合并。LLM 不得删除原问题中明确出现的直接主体，也不得加入仅为背景关联的角色作为核心实体。

### 5.3 删除重复搜索规划

生成阶段不再调用 LLM 工具来重新制定搜索计划。搜索计划统一由问题理解模块产出，并作为外部检索、来源治理、正文生成和延伸阅读的共同输入。

## 6. 分层检索

### 6.1 第一轮检索

第一轮使用两至三条高价值查询，优先覆盖：

1. 官方网站和已验证官方账号；
2. `baike.mihoyo.com` 与 `wiki.hoyolab.com` 等官方运营 Wiki；
3. 游戏内剧情、大世界、武器、圣遗物、书籍、物品和任务文本聚合；
4. 与核心实体直接相关的中性资料页。

第一轮查询必须包含核心实体或精确别名。不得为了扩大召回而只搜索关联角色、组织或宽泛主题。

### 6.2 提前停止条件

满足以下全部条件时，停止扩展搜索并进入正文生成：

- 至少获得三条可用证据；
- 至少两条属于官方来源或高可信资料库；
- 证据直接涉及问题中的核心实体；
- 多实体关系题至少有一条证据同时涉及关系双方；
- 关键事实之间不存在未解决的冲突。

“三条证据”是默认门槛，不是机械要求。若问题本身只有一条唯一官方公告可以直接回答，可由明确的官方单一来源满足正文生成条件；系统需要记录采用单一权威来源的原因。

### 6.3 第二轮检索

第一轮不满足停止条件时，追加一至两条查询，依次考虑：

- Fandom、BWiki 等中性聚合资料；
- 官方社区账号和官方视频；
- 与问题直接相关的普通社区资料。

第二轮结果仍须通过实体门控、问题意图门控和来源治理。社区推测只能说明“存在这种看法”，不能支撑确定性结论。

### 6.4 候选页面补全

搜索结果先经过规则过滤和初步排序，再对可能进入最终证据集的候选页面进行抓取。

- 不再固定补全前十四个页面。
- 默认仅补全排名靠前的五至八条候选。
- 官方域名、官方运营 Wiki 和已具备完整元数据的资料页无需重复抓取发布者身份。
- 社区平台、视频平台和短链接仅在其可能进入正文证据或补充资料时抓取页面。

## 7. 来源审核

### 7.1 规则直接确定的来源

以下来源由可审计规则直接处理，不调用 LLM 内容分类：

- 官方域名；
- 官方运营 Wiki；
- 已登记账号 ID 的官方社区或视频账号；
- 标题、路径和摘要已明确属于公告、角色资料或游戏文本的页面；
- 明确包含配队、武器、圣遗物、输出手法等攻略信号的页面；
- 明确包含猜测、预测、个人理解等推测信号的页面。

### 7.2 需要 LLM 复核的来源

只将以下低确定性候选批量交给 LLM：

- 无法确认作者身份的社区内容；
- 标题和摘要不足以区分中性资料、攻略、考据或推测的页面；
- 规则信号互相冲突的页面；
- 可能进入最终证据集，但内容类型仍不明确的页面。

LLM 只能将来源判为更保守的内容类型，不能提升发布者权威。若 LLM 不可用，规则结果仍可独立运行；不明确来源按较低权威处理。

## 8. 正文生成、校验与修复

### 8.1 主要生成

正文只执行一次主要 LLM 生成。输入包括：

- 原始问题；
- 结构化问题理解；
- 玩家进度与剧透设置；
- 受控语料；
- 最终外部证据；
- 每条来源的权威和内容类型。

模型不得从自身记忆补充没有证据支持的事实。

### 8.2 硬错误

以下问题视为硬错误，可触发一次 LLM 修复：

- 中文问题输出英文或明显中英夹杂；
- 引用 ID 不存在；
- 回答主体偏离核心实体；
- 多实体关系结论缺少共同证据；
- 将社区分析或推测写成官方确认；
- 答案没有回应用户的核心问题；
- 输出无法解析为规定结构。

修复请求只提供失败原因和现有证据，不重新执行搜索。修复最多一次。

### 8.3 软问题

以下问题通过本地规则整理，不调用第二次 LLM：

- 固定连接句使用偏多；
- 段落间存在重复过渡语；
- 引用标记前后空格或标点不统一；
- 可安全删除的套话；
- 不影响语义的轻微格式问题。

若本地整理可能改变事实含义，则不执行该项整理。

### 8.4 修复失败

修复后仍存在硬错误时，放弃风险答案，改用证据拼装式保守回答：

- 先说明能够确认的内容；
- 再说明证据无法确认的部分；
- 每个事实均保留有效引用；
- 不输出推测性结论。

## 9. 流式接口

SSE 事件定义调整为：

### 9.1 `trace`

用于显示简短阶段状态，例如：

- 理解问题；
- 核验资料；
- 整理回答；
- 补充相关资料。

追踪事件应包含服务端时间戳和阶段耗时，供性能验收和回归监控使用。

### 9.2 `answer`

第一阶段完成后发送，包含：

- 正文；
- 结构化段落；
- 正文实际使用的引用；
- 置信度；
- 搜索计划摘要；
- 剧透处理结果；
- 第一阶段耗时。

前端收到 `answer` 后立即结束主要加载状态并恢复输入框。

### 9.3 `resources`

第二阶段完成后发送，包含：

- 延伸阅读；
- 官方视频；
- 补充资料卡片；
- 第二阶段耗时。

`resources` 不得覆盖或改写已经发送的正文。

### 9.4 `done`

所有已启动阶段结束后发送。应标明补充资料阶段是成功、跳过、超时还是失败。

## 10. 前端行为

- 正文到达后立即展示，输入框恢复可用。
- 补充资料尚未完成时，在资料区域显示轻量提示“派蒙还在整理相关资料……”。
- `resources` 到达后静默替换提示并展示资料卡片。
- 补充阶段失败或超时后移除提示，不显示阻断式错误。
- 用户在补充阶段完成前提交新问题时，客户端取消上一请求的剩余读取。
- 服务端应监听请求取消信号，并尽可能终止上一请求尚未完成的补充搜索。
- 取消补充阶段不影响已经展示的正文与引用。

## 11. 缓存

使用进程内短期缓存降低重复网络请求和重复分类成本。

| 缓存内容 | 默认 TTL |
| --- | ---: |
| 页面元数据、重定向结果和发布者身份 | 30 分钟 |
| 搜索结果和来源分类 | 10 分钟 |
| 结构化问题理解 | 30 分钟 |

缓存键至少包含：

- 语言；
- 核心实体及别名；
- 问题意图；
- 查询规范化结果；
- 当前游戏版本上下文。

玩家剧情进度和剧透偏好不会改变页面搜索结果，但会改变最终证据使用和回答内容。因此它们应进入回答阶段的上下文，不能与不同玩家共享最终答案缓存。

本次不对最终答案做长期缓存。进程重启后缓存丢失属于可接受行为。

## 12. 时间预算

普通问题采用以下软预算：

| 阶段 | 目标耗时 |
| --- | ---: |
| 问题理解 | 0–2 秒 |
| 第一轮检索与必要审核 | 4–6 秒 |
| 第二轮检索（按需） | 0–3 秒 |
| 正文生成 | 3–4 秒 |
| 本地校验 | 低于 100 毫秒 |
| 补充资料 | 0–3 秒 |

总预算：

- 正文通常在 8–12 秒内返回；
- 普通请求在 12–15 秒内全部完成；
- 硬错误修复允许额外增加约 3–4 秒；
- 达到 15 秒时若核心证据仍不足，应返回明确的证据不足结果，而不是继续无界等待。

网络超时是保护上限，不等于阶段预算。实现时应使用阶段截止时间和取消信号，使慢请求不能阻塞已经满足条件的快速结果。

## 13. 降级与错误处理

- 第一轮证据充分：停止扩搜，立即生成正文。
- 第一轮证据不足：进入第二轮检索。
- 外部搜索不可用但受控语料充分：基于受控语料回答，并明确来源范围。
- 只有单一权威官方来源：允许回答，但记录单一来源原因。
- 只有社区推测：说明暂无可靠定论。
- 15 秒内仍无足够证据：返回证据不足，不猜测。
- LLM 问题理解不可用：使用规则理解。
- LLM 来源分类不可用：使用保守的规则分类。
- LLM 正文生成不可用：使用证据拼装式保守回答。
- 补充资料失败：保留正文和正文引用，跳过资料卡片。

## 14. 可观测性

每个请求记录以下不含用户隐私正文的性能数据：

- 是否调用 LLM 问题理解；
- 实际执行的查询数量；
- 第一轮和第二轮各自耗时；
- 候选来源数量；
- 页面补全数量；
- 送入 LLM 来源复核的数量；
- 是否满足提前停止条件；
- 正文生成耗时；
- 是否触发修复及失败类型；
- 正文首包时间；
- 补充资料完成时间；
- 缓存命中情况；
- 请求是否被用户取消。

这些数据用于发现后续功能增加造成的延迟回归。默认不记录完整问题文本，继续遵守现有用户授权设置。

## 15. 组件边界

建议形成以下清晰边界：

- `question-understanding`：规则理解、按需 LLM 理解和结果合并。
- `search-planner`：根据统一问题理解生成分层查询。
- `evidence-search`：执行第一轮、第二轮和提前停止。
- `source-assessment`：规则审核、歧义来源 LLM 复核和保守合并。
- `answer-generation`：基于最终证据生成正文。
- `answer-validation`：硬错误检测与软问题本地整理。
- `resource-recommendation`：复用候选来源并按需补搜资料。
- `response-stream`：发送 `trace`、`answer`、`resources` 和 `done`。
- `request-cache`：短期缓存和 TTL 管理。

现有模块可渐进调整，不要求一次性进行无关重构。

## 16. 测试策略

### 16.1 单元测试

- 规则置信度足够时不调用 LLM 问题理解。
- 规则冲突时调用 LLM 并保守合并。
- 搜索计划始终保留核心实体。
- 第一轮满足条件时不执行第二轮。
- 单一权威官方来源可以触发停止。
- 多实体关系题缺少共同证据时不得停止。
- 明确来源不调用 LLM 内容分类。
- 歧义来源会进入批量 LLM 复核。
- 软问题只执行本地整理。
- 硬错误最多触发一次修复。
- 延伸阅读优先复用正文候选来源。
- 缓存键不会跨语言、实体或版本上下文污染。

### 16.2 集成测试

覆盖以下场景：

- 普通角色身份题；
- 角色当前状态题；
- 多实体关系题；
- 深度剧情题；
- 游戏玩法题；
- 游戏内大世界、武器、圣遗物或书籍文本题；
- 官方与社区说法冲突；
- 搜索超时；
- 来源发布者无法确认；
- 生成语言错误；
- 无效引用；
- 连续提问取消上一补充阶段。

### 16.3 性能测试

选择一组固定问题进行冷缓存和热缓存测试。每个问题至少运行三次，记录中位数和最慢一次：

- 正文首包时间；
- 完整完成时间；
- LLM 调用次数；
- 网络查询次数；
- 页面补全次数；
- 最终证据数量；
- 是否发生质量降级。

性能测试必须同时运行现有答案质量断言，不能仅以延迟达标作为通过条件。

## 17. 验收标准

- 普通测试问题的正文首包中位数不超过 12 秒。
- 普通测试问题完整完成中位数不超过 15 秒。
- 首次请求也应满足目标，缓存不得成为唯一性能手段。
- 普通快速通道不再执行独立的 LLM 搜索规划。
- 规则可明确判断时不调用 LLM 问题理解或来源分类。
- 第一阶段和第二阶段不重复搜索相同查询。
- 正文到达后即可继续提问，补充资料异步追加。
- 补充资料失败不影响正文。
- 中文、引用、实体一致性、关系证据和权威措辞测试全部通过。
- 现有自动化测试、类型检查和生产构建全部通过。

## 18. 实施顺序

1. 增加阶段耗时和调用计数，建立优化前基线。
2. 删除生成阶段重复的 LLM 搜索规划。
3. 将问题理解改为规则优先、LLM 按需。
4. 实现两轮检索、证据充足判断和提前停止。
5. 将来源 LLM 分类限制到歧义候选。
6. 复用正文候选来源生成延伸阅读。
7. 拆分 SSE 的 `answer` 与 `resources` 事件。
8. 增加请求取消和第二阶段超时。
9. 增加短期缓存。
10. 完成质量与性能回归测试。
