# 派蒙三千问：第二阶段 5 日执行计划

## 1. 阶段目标

第二阶段把现有的被动版本问答 Demo，升级为面向至冬题设的陪伴式预热 Agent。

本阶段以答辩演示为交付目标：评审应能在 5–7 分钟内走完以下闭环：

```text
进入首页
→ 阅读“今日派蒙小纸条”
→ 选择讲解深度
→ 浏览神之心事件链
→ 查看当前主题关系图
→ 继续向派蒙追问
→ 打开发行洞察
→ 看到本次互动形成的现场增量与发行建议
```

完成标准不是“功能数量多”，而是这个闭环稳定、可信、可解释。

## 2. 已确认的产品决策

### 2.1 主入口

首页以“今日派蒙小纸条”为唯一主入口。现有玩家画像、进度、关注点和剧透偏好保留，但缩为轻量设置，不再占据第一屏主体。

### 2.2 主内容线

新增“愚人众与神之心”至冬预热知识线，覆盖：

- 蒙德至挪德卡莱月之七的神之心确定事件；
- 神之心原持有者、获取者、相关执行官与公开后果；
- 冰之女皇和愚人众收集神之心的已公开动机；
- 武器、圣遗物、大世界文献、物品描述等相关暗示；
- 官方明确、剧情暗示和社区推测之间的边界。

现有“桑多涅 × 枫丹机械体系”语料继续保留，用于自由问答、长尾检索和证据能力展示，但不承担第二阶段的主动预热主线。

### 2.3 内容规模

首版建设约 10–12 个概念、20–24 条中英文受控知识条目。范围是可完整演示的纵向事件链，不扩张为愚人众百科或全剧情百科。

### 2.4 内容治理

采用“编辑策划、Agent 讲解”：

- 编辑决定讲什么、节点如何关联、使用哪些证据；
- Agent 根据玩家画像、进度、语言和剧透偏好组织表达；
- Agent 不自行生成事实节点、关系边或当天主题；
- 所有时间线节点和关系边必须能回溯到受控条目。

### 2.5 核心模块

本阶段只实现四个核心模块：

1. 今日派蒙小纸条；
2. 神之心事件时间线；
3. 当前主题的局部关系图；
4. 发行洞察面板。

“版本卖点雷达”作为首页与主题文案的一部分，不单独开发页面。预热内容生成器、分享卡片、完整运营后台暂缓。

### 2.6 两档讲解深度

| 深度 | 玩家价值 | 内容结构 |
|---|---|---|
| 3 分钟轻剧透 | 建立必要的事件与人物关系 | 确定事件链、关键参与者、轻量关系图 |
| 完整考据 | 查看证据层和争议边界 | 确定事件、相关暗示、来源差异、社区推测 |

两档内容都按玩家最新完成的地区主线锁定剧情节点。新增“挪德卡莱（月之一至月之七）”进度；月之八尚未实装。选择“完整考据”不能越过地区进度锁，也不等于永久授权高风险剧透。

### 2.7 预置主题

实现三个可切换的策划主题：

1. 愚人众为什么收集神之心？
2. 七枚神之心分别经历了什么？
3. 冰之女皇的目标目前有哪些已知与未知？

答辩黄金路径固定演示主题 1。主题 2、3 用于证明内容可轮换，不是单页脚本。

## 3. 范围边界

## 4. 黄金演示路径

### 4.1 默认状态

- 语言：中文；
- 玩家画像：回归玩家；
- 进度：枫丹；
- 剧透偏好：轻度剧透；
- 今日主题：愚人众为什么收集神之心？

### 4.2 演示步骤

1. 首页出现“今日派蒙小纸条”，说明过去各国出现的神之心事件可能是进入至冬题设的重要背景。
2. 选择“3 分钟轻剧透”。
3. Agent 给出简短导览，并突出“事件已经确认”和“动机仍有未知”。
4. 打开事件时间线，依次看到蒙德、璃月、稻妻、须弥、枫丹、纳塔、挪德卡莱节点。
5. 点击一个节点，右侧关系图切换到该节点的局部参与者与关系。
6. 点击建议追问，例如“冰之女皇已经明确说过为什么收集神之心吗？”
7. 进入现有问答链路，查看回答、事实状态、引用和剧透保护。
8. 打开发行洞察页，看到：
   - 本次选择了哪档讲解；
   - 浏览了哪个时间线或关系节点；
   - 追问被归类到哪个理解断点；
   - 系统建议补充哪种 FAQ 或预热内容。

## 5. 内容与证据规格

### 5.1 候选概念清单

研究阶段最终确认 10–12 个概念。初始候选为：

1. 神之心的公开定义与作用边界；
2. 蒙德神之心事件；
3. 璃月神之心交易；
4. 稻妻神之心流转；
5. 须弥神之心事件；
6. 枫丹神之心相关公开信息；
7. 纳塔神之心相关已实装事件；
8. 挪德卡莱月之七火神之心事件；
9. 愚人众执行官与神之心任务；
10. 冰之女皇的公开立场；
11. 神之心与天理的相关暗示；
12. 当前仍未被官方解释的问题。

若某地区没有足够可验证信息，不得为了凑齐结构而补写结论，应明确显示“尚无足够公开证据”。

### 5.2 允许使用的证据

优先级如下：

1. 官方网站、官方公告、官方视频；
2. 正式游戏内文本：
   - 主线、传说任务和世界任务文本；
   - 大世界书籍、文献和场景文本；
   - 武器故事；
   - 圣遗物故事；
   - 物品、材料和任务道具描述；
3. Wiki 或资料站对游戏内文本的索引；
4. 社区考据，仅作为明确标记的社区推测。

Wiki 是检索入口或文本索引，不因承载游戏文本而自动成为官方来源。正式条目应尽量记录原始游戏内文本出处。

### 5.3 禁止使用的证据

- 泄露和未公开测试服信息；
- 数据挖掘但未正式实装的文本；
- 无法定位原始出处的二手结论；
- 将玩家普遍猜测写成剧情暗示；
- 将叙述者观点、寓言或传说直接写成客观事实。

### 5.4 时间线证据规则

- 主时间线只收录 `official_explicit` 的确定事件；
- `narrative_implied` 内容放在节点的“相关暗示”中；
- `community_speculation` 只在完整考据档展示；
- 每个确定事件节点至少绑定一条中英文对应受控条目；
- 每条关系边至少绑定一个来源 ID；
- 无证据关系不得出现在图中。

### 5.5 内容审计表

研究时为每个概念维护以下字段：

| 字段 | 说明 |
|---|---|
| `conceptId` | 跨语言稳定 ID |
| `claim` | 该条目要支持的单一核心陈述 |
| `sourceTitle` | 官方内容或游戏内文本名称 |
| `sourceLocation` | URL、任务名、书籍名、武器名或圣遗物套装名 |
| `sourceKind` | official / game_text / trusted_wiki / community |
| `factStatus` | official_explicit / narrative_implied / community_speculation |
| `spoilerLevel` | 0–3 |
| `minimumProgress` | 最低主线进度 |
| `zh` / `en` | 中英文受控表达 |
| `reviewState` | researched / drafted / reviewed |

## 6. 技术设计

### 6.1 新增领域类型

在 `lib/domain.ts` 增加：

```ts
type PreheatDepth = "guided" | "research";

interface PreheatTopic {
  id: string;
  titleZh: string;
  titleEn: string;
  introZh: string;
  introEn: string;
  heroConceptIds: string[];
  timelineNodeIds: string[];
  relationGraphId: string;
  suggestedQuestionsZh: string[];
  suggestedQuestionsEn: string[];
}

interface TimelineNode {
  id: string;
  region: Progress;
  titleZh: string;
  titleEn: string;
  eventConceptIds: string[];
  implicationConceptIds: string[];
  participantIds: string[];
}

interface RelationNode {
  id: string;
  labelZh: string;
  labelEn: string;
  kind: "person" | "faction" | "object" | "concept";
  conceptIds: string[];
}

interface RelationEdge {
  id: string;
  from: string;
  to: string;
  labelZh: string;
  labelEn: string;
  factStatus: FactStatus;
  conceptIds: string[];
}
```

关系图使用受控静态结构，不在浏览器中临时推断关系。

### 6.2 数据文件

建议新增：

- `data/gnosis-knowledge.ts`：20–24 条双语受控知识；
- `data/preheat-topics.ts`：三个策划主题和两档讲解配置；
- `data/gnosis-timeline.ts`：确定事件节点和相关暗示绑定；
- `data/gnosis-relations.ts`：主题局部关系图；
- `data/preheat-events.ts`：历史匿名互动基线。

`data/knowledge.ts` 导出时合并现有桑多涅语料与新神之心语料，保持现有检索入口不变。

### 6.3 预热编排层

新增 `lib/preheat.ts`，职责为：

- 根据主题、语言、讲解深度和玩家偏好返回预热视图；
- 按剧透偏好过滤节点详情和相关暗示；
- 验证主题、时间线、关系图引用的 `conceptId` 均存在；
- 生成无模型密钥时的确定性讲解；
- 配置模型时，只允许模型基于已选受控条目改写表达。

建议新增接口：

```text
GET  /api/preheat?topicId=...&depth=...&language=...
POST /api/preheat/events
```

`GET /api/preheat` 返回主题、讲解、可见时间线、当前局部关系图和建议追问。
`POST /api/preheat/events` 只接收经过枚举约束的匿名互动事件。

### 6.4 互动事件模型

不要把点击事件伪装成 `QuestionEvent`。新增独立类型：

```ts
type PreheatInteractionKind =
  | "depth_selected"
  | "timeline_node_opened"
  | "relation_node_opened";

interface PreheatInteractionEvent {
  id: string;
  occurredAt: string;
  language: Language;
  playerProfile: Profile;
  topicId: string;
  interactionKind: PreheatInteractionKind;
  targetId: string;
  depth?: PreheatDepth;
  sourceKind: "historical_sample" | "live_increment";
}
```

原始自然语言追问继续使用现有 `QuestionEvent`。洞察聚合接收两类事件：

- `PreheatInteractionEvent` 表示兴趣；
- `QuestionEvent` 表示理解断点。

不得保存页面浏览轨迹、停留时间、鼠标移动或完整会话。

### 6.5 存储

本地 Demo：

- 问题事件继续写入 `.data/events.json`；
- 预热互动写入 `.data/preheat-events.json`；
- 读取时与各自历史基线合并。

Supabase：

- 新增 `supabase/migrations/0002_preheat_interactions.sql`；
- 创建 `preheat_interaction_events` 表；
- 使用服务端 service role；
- 不开放匿名客户端读取原始事件。

### 6.6 洞察聚合

扩展 `lib/insights.ts`：

- 保留现有问题类别、困惑主题、语言差异分析；
- 新增主题兴趣排行；
- 新增讲解深度分布；
- 新增时间线节点兴趣排行；
- 对比“高点击、低追问”和“高点击、高追问”：
  - 高点击、低追问：内容有吸引力且较易理解；
  - 高点击、高追问：内容有吸引力但存在理解断点；
  - 低点击、高追问：入口表达可能不清楚，玩家直接寻求帮助。

第二阶段新增的建议模板至少覆盖：

- 中英文术语或表达差异；
- 高兴趣节点的社媒预热选题。

所有建议仍保持 `draft`，必须显示触发规则和聚合证据。

## 10. 风险与降级方案

| 风险 | 影响 | 降级方案 |
|---|---|---|
| 神之心部分地区证据不足 | 时间线不完整 | 明确显示“公开证据不足”，不补写结论 |
| 研究内容超过 1 天 | 挤压 UI 时间 | 优先完成主题 1 和主时间线，主题 2、3 复用同一条目 |
| 关系图开发超时 | 阻塞 Day 3 | 使用受控 SVG/CSS 固定布局，不引入图谱引擎 |
| 模型或网络现场失败 | 无法生成讲解 | 使用 `lib/preheat.ts` 的确定性讲解与受控语料 |
| 现场增量过小 | 洞察不明显 | 使用历史基线托底，并单独高亮本次新增 1–2 条事件 |
| 事件类型混淆 | 洞察结论失真 | 保持互动事件与问题事件两种独立模型 |
| 剧透档位与现有偏好冲突 | 用户误解授权范围 | 讲解深度只作用于当前主题，高风险剧透仍二次确认 |
| 内容被误认为 7.0 官方爆料 | 课题可信度受损 | 页面持续标注“至冬预热题设”，未确认内容显示为未知 |

## 12. 建议的每日停线规则

为了确保五天内交付，每天结束时按以下规则收口：

- Day 1 未完成证据审计，不开始扩写非核心主题；
- Day 2 API 未跑通，不提前做复杂动效；
- Day 3 黄金玩家路径未跑通，不增加额外卡片或图表；
- Day 4 现场增量未进入洞察，不美化次要后台区域；
- Day 5 发现阻塞问题时，优先保证主题 1 的完整演示，主题 2、3 可降低内容丰富度但不可失效。
